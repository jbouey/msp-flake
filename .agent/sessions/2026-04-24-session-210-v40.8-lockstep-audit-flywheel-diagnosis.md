## Substrate-violation false-positive cleanup (post-prod-recovery audit)

Investigated 4 open `substrate_violations` rows on prod. Three were
false positives caused by real bugs:

1. **`discovered_devices_freshness`** — invariant LEFT JOIN'd raw
   `discovered_devices` rows. There's no UNIQUE(mac, site_id), so
   each MAC had multiple rows (one fresh + several stale). Each stale
   row generated a violation even when a fresh row also existed. Fix:
   GROUP BY (LOWER(mac), site_id), MAX(last_seen_at) before joining.
2. **`legacy_bearer_only_checkin`** (sev3) — both 0.4.10 daemons were
   signing every checkin, but `sigauth_observations` was empty for
   2h+. Root: `signature_auth.py:_nonce_seen` passed the
   f-string `"7200 seconds"` where asyncpg's INTERVAL bind expected a
   `timedelta` — every verification raised inside `interval_encode`
   ("'str' object has no attribute 'days'"), the surrounding
   try/except treated it as "nonce not seen, fall back to bearer,"
   nothing got written to the observations table, and the
   bearer-only invariant fired forever. Fix: pass `NONCE_TTL` (already
   a timedelta) directly. Removed `::interval` cast — asyncpg binds
   timedelta natively.
3. **`l2_decisions_stalled`** (sev2) — fired since 2026-04-12 because
   L1 covers every incident type the fleet sees and L2 LLM was
   correctly never invoked. Pre-fix: invariant treated zero
   l2_decisions as a stall. Post-fix: also requires evidence the
   healing pipeline is running (≥5 incident_remediation_steps in 48h)
   OR ≥3 unresolved incidents older than 30 min with no remediation
   steps (the silent-escalation pattern). Quiet-but-healthy state
   (everything handled by L1) keeps the invariant silent.

The 4th open row (`installer_halted_early` on 84:3A:5B:91:B6:61) is
not a false positive — it's the third t640 appliance, physically alive
but `site_appliances` row was deleted by the cleanup pass earlier
today and `/api/provision/rekey` rejects "unknown appliance." Recovery
needs the box's LAN IP from the operator (queued).

## Hook hardening — class-level rename detection (commit 6af14d4b)

The 12-class agent_api.py rename in 7b3afbc7 had to use
`PYDANTIC_CONTRACT_BREAKING=1` because the contract check treated
each class rename as N field-removal violations. Field-level rename
detection (already present) only handles renames inside ONE class.

New: `_detect_class_renames()` finds classes that disappeared from
the old AST and looks for added classes with IDENTICAL field
signatures (sorted (name, type) tuples). Unambiguous matches (one
new class per old class) are treated as renames and INFORMATIONAL
in the hook output — they don't block the commit. Classes with
empty signatures (no annotated fields) and ambiguous matches (two
new classes share an old class's signature) still surface as
removals so the human reviews.

Tests added (4 new) + existing tests updated for the new
`(violations, renames)` tuple return type from `_diff_models`.
23/23 pass.

## Substrate-violation v2 refinements (post-deploy audit)

After the round-1 substrate fix landed (`e64e1916`), prod re-evaluation
showed two issues:

**Round 2 — `legacy_bearer_only_checkin` re-fix (`06a22794`)**

Round-1 removed the `$2::interval` cast from `_nonce_seen` thinking
asyncpg's typed bind would communicate the parameter type. It does
NOT — Postgres prepares the SQL BEFORE bind, so the planner needs
the type from the SQL. With `created_at > NOW() - $2` and no cast
the planner failed at PREPARE with:

    asyncpg.exceptions.UndefinedFunctionError: operator does not
      exist: timestamp with time zone > interval

Restored the cast; kept the timedelta bind. With the cast back, the
sigauth observation hook ran cleanly and `sigauth_observations`
started receiving rows. The bearer-only invariant auto-resolved.

**Round 2 — `l2_decisions_stalled` re-refinement (`b2d4285e`)**

Round-1 added `pipeline_signal = remediation_steps_48h >= 5` as the
"L1 is firing so L2 should fire too" gate. Prod showed 85 successful
L1 steps (all `result='order_created'`) + 0 L2 decisions — that's
NOT a stall, that's L1-saturation. The gate counted L1 SUCCESSES as
evidence of an L2 stall, which inverted the meaning.

Replaced with: `failed_l1_steps_48h >= 3` where `failed` means
`result NOT IN ('order_created','success','completed','order_acked')`.
A failed L1 step is the natural escalation point to L2. Failures
piling up + zero L2 decisions = real silent stall.

Removed the `silent_escalations` gate (it caught operator-only
incident types which the L2 endpoint correctly short-circuits — those
were never L2-eligible to begin with). Tests updated to match;
test_no_violation_when_l1_covers_everything codifies the correct
quiet-but-healthy state.

**Side-effects of round 2:**

- `discovered_devices_freshness`: cleared (round-1 multi-row dedup fix)
- `legacy_bearer_only_checkin`: cleared (sigauth observations now writing)
- `l2_decisions_stalled`: stops firing on healthy fleet
- `installer_halted_early`: only the orphan (84:3A:5B:91:B6:61) remains
  — pending LAN IP for `recover_legacy_appliance.sh`
- New `signature_verification_failures` opened — daemons ARE signing
  but every signature is `valid=false reason=invalid_signature`.
  Investigation queued as a separate task.
