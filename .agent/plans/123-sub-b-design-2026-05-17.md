## Goal

`POST /api/admin/sites/{site_id}/appliances/revoke-bearers`

Admin-only endpoint that:
1. Validates the caller-supplied `appliance_ids` belong to `{site_id}` (no cross-site spoof)
2. In one `admin_transaction`:
   - Writes a `compliance_bundles` row with `check_type='privileged_access'` + `event_type='bulk_bearer_revoke'` + per-appliance fan-out (via `target_appliance_ids[]` jsonb array — mirrors #118)
   - `UPDATE site_appliances SET bearer_revoked=TRUE WHERE site_id=$1 AND appliance_id = ANY($2::uuid[])`
   - `UPDATE api_keys SET active=FALSE WHERE site_id=$1 AND appliance_id = ANY($2::uuid[])` (Sub-A only registers the privileged-order type; this is where the actual bearer credential gets disabled)
   - Writes `admin_audit_log` row with `action='bulk_bearer_revoke'`, `target=site_id`, denormalized `details` for the substrate invariant's lookup
3. Returns 200 with `{ok, attestation_bundle_id, chain_position, revoked: [{appliance_id, hostname}], already_revoked: [{appliance_id, hostname}]}`

Result: the daemon's next checkin hits 401 via `shared.py:614-640` short-circuit. Recovery path is per-appliance `watchdog_reset_api_key` privileged event (separate Gate A — out of scope here).

## Idempotency (Gate B P1-5)

The runbook's retroactive-attestation procedure depends on idempotency: an operator who already ran `UPDATE site_appliances SET bearer_revoked=TRUE` via psql during an incident should be able to call this endpoint with the same `appliance_ids` and get **a fresh attestation written** (closing the substrate invariant) WITHOUT erroring on the already-TRUE column.

Two distinct cases:

| Case | bearer_revoked | api_keys.active | Expected behavior |
|------|----------------|-----------------|-------------------|
| Fresh revocation | FALSE | TRUE | Flip both + write attestation + audit row |
| Retroactive attestation | TRUE | (any) | Skip column flips, still write attestation + audit row, return `already_revoked: [...]` |
| Partial state (one TRUE, one FALSE in batch) | mixed | mixed | Per-row decision; one attestation bundle covers all `appliance_ids` regardless |

The endpoint partitions `appliance_ids` at lookup-time:
- `to_flip[]` = those with `bearer_revoked=FALSE`
- `already_revoked[]` = those with `bearer_revoked=TRUE`
- Attestation `target_appliance_ids[]` = `appliance_ids` (all of them — the auditor sees the full intent of the operator's call, not the partial-state outcome)
- UPDATEs only fire for `to_flip[]`

## Site/appliance ownership validation

**P0-1 fix:** `appliance_id` is `text`/`character varying` in prod
(verified via `prod_column_types.json` — `site_appliances.appliance_id`
= `character varying`, `api_keys.appliance_id` = `text`).
NOT `uuid`. Using `::uuid[]` would trigger the 2026-05-13
signature_auth.py:618 outage class pinned by
`tests/test_no_uuid_cast_on_text_column.py`. Must be `::text[]` +
`List[str]` in pydantic.

**P0-3 fix:** soft-deleted appliances + unknown appliances both
return 404 with identical body. Existence-oracle attack class
closed — admin sees the distinction in `admin_audit_log.details`
which only the admin context can read.

```python
existing = await conn.fetch(
    """
    SELECT appliance_id, hostname, bearer_revoked, deleted_at
      FROM site_appliances
     WHERE site_id = $1
       AND appliance_id = ANY($2::text[])
       FOR UPDATE
    """,
    site_id, appliance_ids,
)
# Partition: present-and-live vs missing-OR-soft-deleted (Maya P0-3:
# 404 for both, no distinction in user-facing response).
live_rows = [r for r in existing if r["deleted_at"] is None]
live_set = {r["appliance_id"] for r in live_rows}
not_actionable = set(appliance_ids) - live_set  # missing ∪ soft-deleted
if not_actionable:
    # Denormalize the distinction into admin_audit_log.details for
    # forensics (admin context only — not echoed to caller).
    admin_audit_details["not_actionable"] = sorted(not_actionable)
    # P1-pending: write the audit row BEFORE raising so the operator
    # has post-hoc traceability even for failed calls.
    raise HTTPException(
        status_code=404,
        detail="one or more appliance_ids not found at this site",
    )
```

**Gate A P1 fix:** `FOR UPDATE` on the SELECT closes the TOCTOU
window between lookup + UPDATE (mirrors `vault_key_approval_api.py:141`).

## Audit row denormalization (substrate invariant lookup)

**P0-4 fix:** Sev1 `bearer_revoked_without_attestation` invariant
queries `compliance_bundles WHERE check_type='privileged_access'
AND summary::jsonb->>'event_type'='bulk_bearer_revoke' AND
(summary::jsonb->'target_appliance_ids') ? sa.appliance_id::text`
(see `assertions.py:2495`). The endpoint calls
`create_privileged_access_attestation(conn, site_id=..., event_type=
"bulk_bearer_revoke", approvals=[{"target_appliance_ids":
appliance_ids, ...}])` — and `privileged_access_attestation.py`
stores `event_type` + `approvals` under the `summary` jsonb column
of `compliance_bundles` (NOT `parameters`). The original v1 design's
prose said "parameters" — that was a documentation defect; the
implementation has always used `summary` because the attestation
module is the single writer.

The `admin_audit_log` row is a separate operator-visibility surface;
both writes happen in the same `admin_transaction`.

## Request schema

**P0-1 fix:** `appliance_ids` typed as `List[str]` (not `List[UUID]`) —
matches the prod `text`/`varchar` column types. Regex-validated for
basic UUID-shape sanity (without forcing the `uuid` cast that
breaks the column-type-mismatch class).

```python
_APPLIANCE_ID_RE = re.compile(r"^[a-fA-F0-9-]{32,40}$")

class RevokeBearersRequest(BaseModel):
    appliance_ids: List[str] = Field(min_length=1, max_length=50)
    actor_email: EmailStr
    reason: str = Field(min_length=20, max_length=1000)
    incident_correlation_id: Optional[str] = None

    @field_validator("appliance_ids")
    @classmethod
    def _check_shape(cls, v: List[str]) -> List[str]:
        for aid in v:
            if not _APPLIANCE_ID_RE.match(aid):
                raise ValueError(f"appliance_id has wrong shape: {aid!r}")
        return v
```

Why max=50? Mirrors the #118 fan-out cap. A 250-appliance fleet means a 5-call sequence — operator confirms each in turn rather than one nuclear-button. If a future engagement needs full-fleet revocation, that's a separate `bulk_bearer_revoke_partner_wide` privileged event with its OWN Gate A (mirrors #124's `--all-at-partner` split).
