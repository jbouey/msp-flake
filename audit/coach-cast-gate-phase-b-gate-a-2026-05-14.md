## 250-WORD SUMMARY

**Verdict: BLOCK Phase B as scoped. Approve a re-scoped Phase B-lite instead.**

Phase B as briefed is built on three numbers that the evidence contradicts. (1) The brief says "126 `$N::TYPE` casts" — actual count is **444**. (2) It assumes most casts are `col = $N::TYPE` resolvable against a column — actual gate-relevant `col = $N::T` shape is **~195**; the other **~249** sit in VALUES lists, `jsonb_build_object` args, bare function args, and `interval`/`int` literals where **there is no column to resolve against at all** — sqlparse cannot help them. (3) It assumes the JOIN-ambiguity skip class is ~15% — sampling the three largest SQL files (routes/agent_api/evidence_chain, 174 `text("""` blocks) found **zero** multi-table `::uuid` casts. The hard cases the sqlparse machinery exists to handle barely occur.

Meanwhile **sqlparse is not in `requirements.txt` / `requirements.lock` and is not importable in the venv** — Phase B needs a new prod dependency + Dockerfile rebuild (the manual-rebuild class from CLAUDE.md).

The fixture augmentation is real work with a real blast radius: **4 consumers** all assume `dict[str, list[str]]` and **all 4 break atomically** on a shape change, plus the regen SQL snippet must change.

Coach + PM concur: Phase A already closed the **demonstrated outage class** (the 6 TEXT columns). Phase B's incremental catch is ~189 additional `col=$N::T` casts with **near-zero historical incidence**, at ~4–5h cost including a new prod dep. Defer the full sqlparse build. Ship **Phase B-lite**: augment the fixture with types (high standalone value, unblocks future work) + extend the *existing regex* gate to all `col = $N::T` pairs vs the typed fixture. No sqlparse, no new dep, ~1.5h, catches the same incident class as the full build for the shapes that actually exist.

## PER-LENS VERDICT

### 1. Steve (Principal SWE) — BLOCK

**sqlparse is not installed.** Not in `requirements.txt`, not in `requirements.lock`, not importable in the backend venv. Phase B as scoped requires adding a new **production** dependency (the gate runs in CI which builds against `requirements.lock`). Per CLAUDE.md, new Python deps require a manual `scp` of Dockerfile + requirements.lock to the VPS and a `docker compose build` — this is not a free `pip install`. A CI-only test dependency *could* be added to a test-only requirements file, but the brief didn't scope that and the project doesn't currently have that split for the backend.

**The parse surface is bigger than briefed.** 362 `text("""` blocks across 22 files (routes.py alone has 72). Plus asyncpg raw-string queries (`conn.fetch("...")`) which are NOT `text("""`-wrapped — the brief's extraction step (a) only named `text("""...""")` and "asyncpg raw query strings" loosely; enumerating the asyncpg surface is itself unscoped work.

**The cast taxonomy defeats the premise.** Of 444 `$N::TYPE` casts: ~195 are `col = $N::T` (resolvable), but **~249 are not** — VALUES tuples (`VALUES ($1::uuid, $2, $3, $4::jsonb)`), `jsonb_build_object` args, `compute_partner_rate_bps($1::uuid, $2::int)` function args, `NOW() - $1::interval`. For these there is **no column on the other side of an `=`** — sqlparse parsing them tells you nothing because there's nothing to look up. The sqlparse walker only earns its complexity on the `col = $N::T` subset, and that subset is regex-tractable (Phase A already proved it).

**Column→table resolution is the genuinely hard part and it barely occurs.** Sampling the 3 largest SQL files for multi-table (`≥2 JOIN/FROM`) `::uuid` casts returned **zero hits**. The "~15% JOIN skip class" the brief inherited from Phase A's Gate A is, on inspection, closer to ~0% for the cast-relevant shape. sqlparse's value proposition — robust parsing of gnarly multi-JOIN SQL — is being bought to solve a problem the codebase doesn't have.

**Verdict:** the sqlparse approach is the right tool for a problem that isn't here. BLOCK; re-scope to extend the proven regex against a typed fixture.

### 2. Maya (Database) — BLOCK on consumer blast radius; APPROVE fixture augmentation as standalone

**Current fixture:** `dict[str, list[str]]` — 311 tables, name-only, 97KB. Generated by an inline SQL snippet documented in `test_sql_columns_match_schema.py` L20-45 (`json_agg(column_name ORDER BY ordinal_position)` from `information_schema.columns`).

**Augmentation shape:** to add types, the value must become either `dict[str, dict[str,str]]` (`{col: data_type}`) or `list[[col,type]]`. Either way it is **not** backwards-compatible with the current `list[str]` shape. Fixture roughly doubles to ~180–200KB — acceptable.

**Four consumers, all break atomically:**
- `test_sql_columns_match_schema.py` — L98 `{tbl.lower(): {c.lower() for c in cols} ...}` — iterates values as a flat string list. **Breaks.**
- `test_migration_view_columns_exist.py` — L192-193 `{tbl: set(cols) ...}`. **Breaks.**
- `test_no_uuid_cast_on_text_column.py` (Phase A) — currently uses a hardcoded `_TEXT_COLUMNS` frozenset, **does not actually read the fixture yet** despite its docstring claiming to. So it doesn't break — but Phase B's whole point is to make it read the fixture.
- `test_no_raw_discovered_devices_count.py` — L89-93 only checks key presence (`"canonical_devices" in fixture`). Survives a value-shape change **if** keys stay table names. Low risk but must be verified.

Plus the **regen SQL snippet** in `test_sql_columns_match_schema.py` must change (`json_agg(json_build_array(column_name, data_type) ...)` or similar) and be re-run against prod — otherwise the next person who refreshes the fixture silently reverts the shape.

**This is a 5-artifact atomic change** (fixture + regen doc + 2 hard consumers + verify 1 soft consumer), not the "~30min 3-consumer-update" the brief estimated. The brief also referenced `test_data_model_audit_contract.py` as a consumer — it exists but does **not** grep-match `prod_columns`, so it is not actually a consumer; the brief's consumer list was inaccurate in both directions.

**The fixture augmentation itself is good and worth doing** — a typed schema fixture has standalone value for future gates. But it must land as its own atomic commit with all consumers migrated in lockstep, and it does not require sqlparse.

## RECOMMENDATION: DEFER full Phase B — implement Phase B-lite instead

**Rationale:** Phase A closed the demonstrated outage class. Phase B's full sqlparse build buys comprehensiveness over a near-zero-incidence tail, costs ~5h + a new prod dependency, and the column→table resolution that justifies sqlparse over regex effectively does not occur in this codebase (0 multi-JOIN cast callsites in the 3 largest files). The fixture augmentation, however, is reusable infrastructure worth doing now.

### Phase B-lite sub-plan (~1.5h, APPROVED to implement)

**Step 1 — Augment `prod_columns.json` with types (~55min).**
- Update regen SQL snippet in `test_sql_columns_match_schema.py` to emit `{table: {column: data_type}}`.
- Re-run against prod (`ssh root@178.156.162.116 ...`), commit the typed fixture.
- Migrate all 4 real consumers in the **same commit**: `test_sql_columns_match_schema.py` (L98 + regen doc), `test_migration_view_columns_exist.py` (L192-193), `test_no_raw_discovered_devices_count.py` (verify only), and wire `test_no_uuid_cast_on_text_column.py` to read the fixture.
- Run the curated pre-push sweep + the 4 consumer tests; cite pass count at Gate B.

**Step 2 — Extend the existing regex gate to all `col = $N::T` pairs (~35min).**
- In `test_no_uuid_cast_on_text_column.py` (rename to `test_no_param_cast_against_mismatched_column.py` per task #77 title): replace the hardcoded `_TEXT_COLUMNS` frozenset with a fixture-driven check — for every `(?:tbl\.)?col\s*=\s*\$\d+::(\w+)` match, resolve `col` against the typed fixture and flag when the cast type ≠ column type (with a type-equivalence map: `uuid`/`text` distinct, `int`/`bigint`/`integer` equivalent, etc.).
- Ambiguity handling without sqlparse: when a bare `col` name maps to multiple tables with **conflicting** types, skip with a documented `# ambiguous-bare-column` carve-out (this is the honest, documented scope reduction — and per the sampling it will be rare). Table-qualified refs (`d.appliance_id`) resolve cleanly via the qualifier when the alias→table map is derivable, else skip.
- Keep the existing 6-column hard pin as a ratchet floor so Phase A's guarantee never regresses.

**Step 3 — Gate B (pre-completion fork) + run full sweep.** Per CLAUDE.md Session 220 lock-in, Gate B must execute the full source-level sweep, not just review the diff.
