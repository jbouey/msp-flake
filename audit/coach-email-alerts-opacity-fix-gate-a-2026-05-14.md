## 250-word summary

`email_alerts.py` is a mixed-recipient module — exactly why Session 218 Phase 0
deferred it. The 4 `msg["Subject"]` interpolation sites split cleanly:

- **L523 `send_critical_alert`** — recipient is `ALERT_EMAIL`
  (`administrator@osiriscare.net`), a hard-coded operator inbox. `title` is
  generated by 16+ internal callers (health_monitor, chain_tamper_detector,
  retention_verifier, etc.) and routinely contains site_id/host_id. **EXEMPT —
  operator channel.** No change.
- **L826 `send_companion_alert_email`** — recipient is a "companion" user
  (external clinic-adjacent reviewer). Subject leaks `org_name` + `module_label`.
  **VIOLATION — must go opaque.**
- **L1165 `send_partner_weekly_digest`** — recipient is partner `contact_email`
  (external MSP). Subject leaks `partner_brand` + `week_label`. `partner_brand`
  is the *recipient's own* brand, not a third-party clinic name — lower
  sensitivity, but still an unauthenticated-channel context leak per Rule 7
  (forwarded/mistyped address). **VIOLATION — must go opaque.**
- **L1287 `send_consent_request_email`** — recipient is the clinic practice
  manager (`for_email`). Subject leaks `partner_brand` + `class_display_name`.
  Fully customer-facing. **VIOLATION — must go opaque.**

Because L523 stays verbose, `email_alerts.py` **cannot** join `_OPAQUE_MODULES`
(that would trip `test_operator_modules_are_not_in_opaque_scope` and force L523
opaque too). The fix is **per-callsite opaque rewrites of the 3 customer-facing
subjects** while the module stays in `OPERATOR_ALLOWLIST`. The deliberate-exclusion
comment is rewritten — not deleted — to record the per-callsite reasoning. A new
narrow pin test (`test_email_alerts_customer_facing_subjects_opaque`) guards the
3 rewritten subjects by line/function so a future edit can't re-leak.

## Per-callsite analysis

### L523 — `send_critical_alert` — **EXEMPT (operator)**

```python
msg["Subject"] = f"[CRITICAL] {title}"
msg["To"] = ALERT_EMAIL          # administrator@osiriscare.net (env, hard default)
```

**Steve:** `title` is the first positional arg. Grepped all 16 callers
(`health_monitor.py`, `chain_tamper_detector.py`, `retention_verifier.py`,
`background_tasks.py`, `healing_sla.py`, `sites.py`, `routes.py`,
`audit_package_api.py`, `client_portal.py`, `partner_auth.py`). Titles are
operator-diagnostic strings — e.g. `"Mesh isolation: {site_id}"`,
`"Chain tamper detected"`, `"Retention gap on {site_id}"`. They routinely embed
site_id/host_id. Recipient is a fixed internal address; there is no
forwarded/mistyped-address threat — the operator inbox is the audience.

**Counsel (Rule 7):** Rule 7 governs *unauthenticated* channels that reach
*customers / outside parties*. The operator alert inbox is an internal
administrative channel. Forcing it opaque would actively degrade incident
response with zero privacy gain. **EXEMPT — no change. Stays in
`OPERATOR_ALLOWLIST`.**

### L826 — `send_companion_alert_email` — **VIOLATION → opaque**

```python
msg["Subject"] = f"[OsirisCare] Compliance Alert: {module_label} overdue for {org_name}"
msg["To"] = to_email             # companion_email — external reviewer
```

**Steve:** recipient `to_email` comes from `companion.py:1438`
`to_email=alert["companion_email"]` — a companion (clinic-adjacent compliance
reviewer) user's address. `org_name` is the *client clinic name*. `module_label`
is a compliance module name. Both are third-party identifying context in an
unauthenticated channel.

**Carol:** this is the textbook leak class — clinic-name correlation in a
subject line that sits in a mailbox / mail-server logs / mobile lock-screen
preview. A forwarded or mistyped companion address exposes "Clinic X is behind
on compliance."

**Proposed opaque subject:**
```python
msg["Subject"] = "[OsirisCare] A compliance module needs your review"
```
Body already says "Please review this client's progress in the Companion
Portal" — identifying context (org_name, module) stays in the body *and* portal.
Per the canonical pattern, body context is acceptable for a logged-in-portal
redirect; **but** the body here is delivered in the email itself. Coach note
below tightens this.

**Coach:** the 3 canonical opaque modules (`client_owner_transfer`,
`client_user_email_rename`, `cross_org_site_relocate`) put **zero** org/clinic
names in the *body* too — body redirects to the authed portal. For strict
parity, the companion email body should also drop `org_name` / `module_label` /
`expected_status` / `current_status` and redirect to the Companion Portal.
**However** — see Counsel middle-ground below — a fully contextless alert email
is arguably useless. Recommended: **subject opaque (mandatory)**; **body may
retain module_label but NOT org_name** (the clinic identity is the sensitive
correlation; the module name alone is low-risk generic-compliance vocabulary).
This requires dropping `org_name` from the body template at L839 + `_send`
desc — a signature change (`org_name` param can stay for audit-logging but must
not reach subject/body), OR drop it entirely. **Recommend: drop `org_name` from
subject AND body; keep `module_label` in body only.**

### L1165 — `send_partner_weekly_digest` — **VIOLATION → opaque**

```python
msg["Subject"] = f"{partner_brand or 'OsirisCare'} · Week in review · {week_label}"
msg["To"] = to_email             # partner contact_email — external MSP
```

**Steve:** `partner_brand` is the *recipient partner's own* brand name (from
`partners.brand_name`), not a third-party clinic. `week_label` is a date range.
Lower sensitivity than L826 — the recipient already knows their own brand.

**Counsel (Rule 7 middle-ground):** Rule 7's threat model is *context the
recipient shouldn't see leaking to a wrong/forwarded address*. A partner's own
brand name in a subject sent to that partner's own contact address is weak as a
leak — but the asymmetry argument still bites: once `client_owner_transfer` et
al. ship fully opaque, a verbose sibling is *attackable as inconsistency*, and a
forwarded digest does reveal "OsirisCare runs compliance for [partner]". Verdict:
**rewrite to opaque** — the bar is "no context name in subject," and
`partner_brand` is a context name. The body keeps full brand + per-client detail
(partner is the authenticated audience for the digest content).

**Proposed opaque subject:**
```python
msg["Subject"] = "Your weekly book-of-business summary"
```
Body unchanged — partner-facing digest content is fine in-body (partner is the
legitimate audience; this is not clinic PHI). `week_label` may be retained in
the body header (already is, L1084).

### L1287 — `send_consent_request_email` — **VIOLATION → opaque**

```python
msg["Subject"] = f"{partner_brand or 'OsirisCare'} · Consent requested for {class_display_name}"
msg["To"] = to_email             # for_email — clinic practice manager
```

**Steve:** recipient `to_email` = `for_email` from `partners.py:1821`, the
practice-manager address the partner typed in. Fully customer-facing —
*and* a magic-link email (the link in the body is a single-use auth token).
Subject leaks `partner_brand` (the MSP's name) + `class_display_name` (the
remediation category).

**Carol:** this is the highest-sensitivity of the four — a mistyped practice-
manager address gets a subject saying "[MSP brand] · Consent requested for
[category]", correlating an MSP to a clinic to a specific remediation class, in
an unauthenticated channel, attached to a live auth token.

**Counsel:** magic-link emails are precisely the RT21 threat model. Subject
**must** be opaque. The body legitimately needs `partner_brand` + category +
the magic link (that's the whole point of the email and it's gated by the
single-use token + email-match check at L1257–1259) — so body stays verbose;
**only the subject** is rewritten. This matches `cross_org_site_relocate`'s
own magic-link pattern (opaque subject, link in body).

**Proposed opaque subject:**
```python
msg["Subject"] = "[OsirisCare] Authorization requested for your practice"
```

## Counsel Rule 7 middle-ground assessment

**"How opaque is too opaque for an actionable alert?"**

The canonical 3-module pattern (zero context in subject *and* body, pure portal
redirect) is correct for **state-transition notifications** (owner transfer,
email rename, relocate) — those emails exist only to say "go authenticate and
look." But two of these four are different:

- **`send_consent_request_email`** is a *magic-link* email — the body MUST carry
  the link and enough context to make an authorization decision. Stripping the
  body would break the feature. Resolution: **opaque subject, verbose body** —
  the body is acceptable because the link is single-use, expires in 72h, and is
  email-address-matched (L1257–1259). This is exactly how
  `cross_org_site_relocate`'s own magic-link email is built. Not too opaque.

- **`send_partner_weekly_digest`** body is digest *content* for the partner
  (their own book of business) — the partner is the legitimate authenticated-
  equivalent audience. **Opaque subject, verbose body.** Not too opaque.

- **`send_companion_alert_email`** is the genuine tension: a pure
  "something needs review, log in" email risks being ignored. Middle ground:
  **subject fully opaque; body keeps the generic `module_label` but drops the
  clinic `org_name`.** `module_label` is generic compliance vocabulary ("Risk
  Assessment", "Backup Verification") — not identifying. `org_name` is the
  clinic identity — the actual Rule 7 leak. This keeps the email actionable
  ("your Risk Assessment module needs review — log in") without naming the
  clinic in an unauthenticated channel.

**Conclusion:** the middle ground is **opaque subjects everywhere (non-
negotiable); body opacity is graded by recipient-legitimacy** — verbose body OK
when the recipient is the authenticated-equivalent audience (partner digest) or
the body is the feature (magic link); body must drop *clinic-identifying* names
(`org_name`) when the recipient is a third party (companion). This is consistent
with Counsel Rule 7 ("context belongs behind auth") — the subject line is the
unauthenticated-by-definition surface; the body's sensitivity depends on who
legitimately receives it.

## Auditor (OCR) — §164.530 administrative safeguards

§164.530(c) requires reasonable safeguards against incidental disclosure. An
opaque-by-default posture for customer-facing email subjects is consistent
administrative-safeguards practice — and *inconsistency* across sibling email
classes is itself an audit finding ("why is this email opaque but that one
isn't?"). Approving the 3 rewrites brings `email_alerts.py` into line with the
3 already-harmonized modules. The operator carve-out for L523 is defensible —
internal administrative communications are not the incidental-disclosure
surface §164.530(c) targets. **Auditor: APPROVE.** Recommend the rewritten
test comment explicitly cite "operator administrative channel" for L523 so a
future auditor sees the reasoning inline.
