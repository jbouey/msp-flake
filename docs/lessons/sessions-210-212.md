## Session 210-B — Appliance relocation, daemon v0.4.11/0.4.12, pydantic contract gate

- **Pydantic dup baseline = 0 (Session 210-B 2026-04-25).** `test_no_new_duplicate_pydantic_model_names` EXPECTED_MAX = 0. Adding any new duplicate `__name__` across BaseModel subclasses fails CI. Class renames with unchanged field signature are auto-permitted by `pydantic_contract_check.py` (informational `RENAMED` line in pre-commit output) — wire-contract-safe since FastAPI's OpenAPI $ref key is the only thing that changes. `test_schema_is_deterministic` passes (no longer xfail). Determinism check is two-run: first run may write if Python version drifted (CI 3.11 vs dev 3.14), second run must say `unchanged`.
- **Migration 244 (Session 210-B):** `discovered_devices` UNIQUE(LOWER(mac), site_id) + 3 GC funcs (`prune_install_sessions(30d)`, `prune_nonces(4h)`, `prune_discovered_devices(60d)`). Wired into `data_hygiene_gc_loop` (24h cadence, 15-min startup delay). `discovered_devices_archive` preserves the dedup snapshot.
- **Migration 245 (Session 210-B): `relocations` table** is the append-only tracker for admin-initiated appliance moves. Columns: source_*, target_*, mac_address, status (pending/completed/expired/failed), reason, actor, fleet_order_id, evidence_bundle_id. UNIQUE(mac, status) blocks concurrent pending moves. `prevent_relocations_delete()` trigger blocks DELETE. `finalize_pending_relocations()` SQL func: target.last_checkin > initiated_at → 'completed' + soft-delete source + deactivate source api_keys; >30 min initiated_at → 'expired'. `relocation_finalize_loop` runs every 60s.
- **Migration 246 (Session 210-B):** `site_appliances.status` CHECK extended with `'relocating'` + `'relocated'`. Migration 245's relocate flow needs both; the check originally only allowed pending/online/offline/auth_failed/decommissioned.
- **Appliance relocate flow (Session 210-B RT-1..8).** `POST /api/sites/{site_id}/appliances/{appliance_id}/relocate` body `{target_site_id, reason}` (reason ≥20 chars). Same-org only (cross-org returns 403, points at future `/api/admin/cross-org-relocate`). Atomic txn in `admin_connection`: validate → UPSERT target site_appliances → mint api_key → mark source `'relocating'` (NOT eager soft-delete) → if `agent_version >= 0.4.11` issue signed `reprovision` fleet_order via `sign_admin_order(target_appliance_id=...)` else return ssh_snippet → write `relocations` row → emit `appliance_relocation` compliance_bundle (admin_initiated variant via `appliance_relocation.emit_admin_relocation_bundle`) → admin_audit_log. Frontend `MoveApplianceModal.tsx` renders fleet-order receipt OR copy-paste ssh_snippet. **Demoed end-to-end on 2026-04-25:** 89s elapsed initiate → completed (orphan moved physical-appliance-pilot-1aea78 → north-valley-branch-2 with no SSH).
- **Daemon `reprovision` order (v0.4.11+, Session 210-B).** `appliance/internal/orders/processor.go::handleReprovision`. Atomic config.yaml rewrite via tmpfile + rename, backup `/var/lib/msp/config.yaml.bak.reprovision.<unix>`, refuses api_keys < 40 chars, ACK-before-restart via 3s-delayed goroutine. Required params: `new_site_id`, `new_api_key`. Optional: `new_host_id`, `restart_delay_sec`. **TODO v0.4.12:** add `reprovision` to `dangerousOrderTypes` map (currently scoped to update_daemon, nixos_rebuild, healing, etc.) — task #164.
- **`relocation_stalled` substrate invariant (sev2, Session 210-B RT-4).** Fires when a `relocations` row stays `pending` for >30 min. Substrate runbook at `substrate_runbooks/relocation_stalled.md` walks the operator through the diagnostic ladder (daemon online? agent_version ≥ 0.4.11? fleet_order_id present? ssh_snippet was returned for legacy daemons?). Total invariant count after this addition: 33.
- **`/api/provision/admin/restore` (Session 210-B).** Admin Bearer-authed endpoint that UPSERTs site_appliances + mints fresh api_key + audit-logs in one txn. Built for the orphan class where the standard rekey endpoint refuses ('unknown appliance' because the row was deleted). Reason ≥20 chars enforced. Pairs with `recover_appliance_full.sh` script.
- **Class-level Pydantic rename detection in pre-commit (Session 210-B).** `scripts/pydantic_contract_check.py::_detect_class_renames` finds classes that disappeared from old AST + signature-match against added classes (sorted (name, type) tuples). Unambiguous matches log informational `RENAMED` line, don't block. Class rename + ANY field type change still surfaces as removals (signature differs). Two-class-same-signature is ambiguous — surfaces removals, no rename claim.
- **Pre-push lifespan-imports guard (Session 210-B).** `tests/test_lifespan_imports_resolve.py` walks main.py's AST, finds every `from dashboard_api.X import Y` inside `lifespan()`, asserts every Y is a top-level binding in X. Catches the deleted-symbol-deferred-import class that crashlooped prod earlier today (l2_auto_candidate_loop). AST-only — no asyncpg/pynacl/sqlalchemy required, runs cleanly in pre-push.
- **Substrate prod-snapshot fixture pattern (Session 210-B).** `tests/fixtures/substrate/<INV>/<scenario>.json` + `tests/test_substrate_prod_fixtures.py`. Captures real prod-shaped DB rows + expected violation count, replays through the assertion. Catches false positives at PR time. Add fixtures liberally — they're cheap, false-positive regressions are expensive.
- **DELETE-impact-check tool (Session 210-B).** `scripts/db_delete_safety_check.py --table site_appliances --where "appliance_id='X'"` runs an impact analysis against IMPACT_MAPS (api_keys, compliance_bundles, incidents, fleet_orders, install_sessions, discovered_devices). `--execute` requires `--reason` ≥20 chars + types-the-key confirmation + writes admin_audit_log. Mandatory before any DELETE on site_appliances/sites/api_keys.
- **`migrate.py` ON CONFLICT DO NOTHING (Session 210-B fix in 1eb7772c).** apply_migration's INSERT into schema_migrations didn't have ON CONFLICT, so migration 000 (which self-INSERTs its own row) collided. Latent forever in long-lived prod DBs (already had the row); blocked every fresh-DB initialization.
- **Sigauth interval bind (Session 210-B fix in 06a22794).** `signature_auth.py::_nonce_seen` MUST keep the `$2::interval` cast in the SQL even when binding a Python timedelta — Postgres needs the cast at PREPARE time before bind to resolve operator types. Without it: `UndefinedFunctionError: operator does not exist: timestamp with time zone > interval`. Removing the cast was wrong; the original f-string-as-text bind was also wrong. Right answer: timedelta bind + ::interval cast.
