## Session 213 — Flywheel canonical aliasing + rename_site() + F7 diagnostic

- **Migration 254 + 255 (Session 213).** 254 retries the orphan `aggregated_pattern_stats` cleanup as a full Step 1+2+3 (merge → delete-collisions → rename) — 252 left 115 orphan rows because asyncpg simple-query / explicit BEGIN/COMMIT skipped sub-statements after the first. 255 closes the **upstream recreation cycle**: every 30 min `_flywheel_promotion_loop` aggregates `execution_telemetry` by `site_id`, so cleanups land then orphan rows get RECREATED on the next tick. 255 relocates `execution_telemetry` (19,063 rows), `incidents` (533), `l2_decisions` (31) from `physical-appliance-pilot-1aea78` → `north-valley-branch-2`, then re-runs the `aggregated_pattern_stats` cleanup. **INTENTIONALLY skips `compliance_bundles`** (137,168 rows — Ed25519 + OTS-anchored; `site_id` is part of the cryptographic binding; auditor kits reconcile via the appliance relocation event).
- **`PhantomSiteRolloutError` precondition (Session 213 F2 P0).** `flywheel_promote.safe_rollout_promoted_rule` now refuses `scope='site'` when `SELECT 1 FROM site_appliances WHERE site_id=$1 AND deleted_at IS NULL LIMIT 1` returns 0 rows. Without it, every future relocate/decom was a latent **phantom-promotion**: the rollout would issue fleet_orders into a dead site and silently succeed. `routes.py::promote_pattern` translates to HTTP 409 with a structured remediation body pointing the operator at `admin_audit_log` for the relocate trail. `scope='fleet'` iterates live sites already so it doesn't need the gate.
- **`flywheel_orphan_telemetry` sev1 substrate invariant (Session 213 F3 P0).** Catches the upstream class that mig 255 closed manually: `execution_telemetry` rows landing under a `site_id` with no live appliances. Threshold: > 10 orphan rows per site_id in 24h. Wired into `ALL_ASSERTIONS` + `_DISPLAY_METADATA` + `substrate_runbooks/flywheel_orphan_telemetry.md` (three-list lockstep). Total invariant count: **45 → 46**. Runbook discriminates "known dead site (rename/decom — auto-relocate)" from "unknown site_id (escalate — possible spoofed appliance)."
- **Three layered flywheel bugs identified (Session 213 round-table verdict, NEEDS-IMPROVEMENT 7 findings):** (1) **Eligibility fragmentation** — evidence trapped under a dead site_id never reaches canonical promotion candidates [F1 shipped + mig 256]. (2) **Phantom void promotions** — rollouts to dead sites silently succeed [F2 shipped]. (3) **Recreation cycle** — cleanup futile without upstream fix [F3 shipped + mig 255]. P1 follow-ups: F4 centralize site rename behind SQL function + CI gate [shipped + mig 257]; F5 deprecate duplicate `_flywheel_promotion_loop` in `background_tasks.py` [**already closed in Session 210-B** — `test_flywheel_reconciliation_orphan_symmetry.py::test_l2_synthetic_ids_excluded_from_platform_aggregation` pins absence; the round-table flag was a duplicate-finding]. - **`canonical_site_id()` STABLE function (Session 213 F1 P0, mig 256).** Resolves any `site_id` to its canonical via `site_canonical_mapping` chain (WITH RECURSIVE, depth-bounded 16, returns input on no-mapping). CHECK constraints: actor must match email regex (CLAUDE.md privileged-access rule, no 'system'/'migration:NNN'), reason ≥20 chars, no-self. Use everywhere telemetry/operational queries aggregate by site_id; **NEVER for `compliance_bundles`** (Ed25519 + OTS bind to original site_id forever — CI gate `tests/test_canonical_not_used_for_compliance_bundles.py` enforces ±5-line proximity check). `_flywheel_promotion_loop` Step 1 + Step 3 in `main.py` use canonical_site_id() in CTE-wrapped SELECTs (single function call per row; planner-cache opportunity, not guarantee).
