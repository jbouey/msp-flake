# Master BAA v2.0 — Hardening Prerequisites Checklist

**For:** Outside HIPAA counsel + OsirisCare engineering, drafting `MASTER_BAA_v2.0`
**Status:** Living checklist — consult BEFORE drafting/editing any v2.0 article or exhibit
**Supersedes:** nothing — this is a new gate doc (Task #70, 2026-05-14)

> v1.0-INTERIM (`MASTER_BAA_v1.0_INTERIM.md`) is the HIPAA-core instrument.
> v2.0 adds outside-counsel commercial/legal hardening (term, termination,
> indemnity, audit rights, governing law). This checklist enumerates
> **engineering-evidence preconditions** that gate specific v2.0 language —
> a v2.0 sentence asserting a capability MUST NOT ship until the matching
> precondition below is satisfied with attached runtime evidence.

This exists because of a near-miss class, not a past breach: a contract
sentence that asserts a capability becomes an over-claim *the moment it is
written* if the capability is not operationally proven. v1.0-INTERIM was
checked and does **not** over-claim (see Task #70 Gate A,
`audit/coach-baa-d1-soak-gate-gate-a-2026-05-14.md`); these gates keep
v2.0 in the same state.

## PRE-1 — Per-event / heartbeat-level verification language

**Gate:** v2.0 hardening MUST NOT introduce per-event, per-heartbeat, or
"continuously verified" language in Article 3.2, Exhibit B, or anywhere
else **unless** D1 heartbeat-signature backend verification has a
≥7-day clean operational soak.

**Why:** D1 (backend verification of daemon-signed heartbeats) was
deployed but inert in prod for 13 days (2026-04-30 → 2026-05-13, fixed
in `adb7671a`). v1.0-INTERIM survives this because it never references
heartbeats — every "cryptographically signed" claim scopes to **evidence
bundles** (`compliance_bundles`), which were Ed25519-signed + OTS-anchored
continuously. A v2.0 sentence reaching for §164.308(a)(1)(ii)(D)
information-system-activity-review strength via "every heartbeat is
verified" would be an over-claim until the soak proves the path.

**Soak-clean bar (cite actual SQL output in the v2.0 sign-off):**
- Trailing **7 consecutive days**, per pubkeyed appliance
  (`site_appliances.agent_public_key IS NOT NULL`):
  `COUNT(*) FILTER (WHERE signature_valid IS TRUE) / COUNT(*) >= 0.99`.
  Per-appliance floor, **not** fleet aggregate (Counsel Rule 4 —
  orphan coverage).
- Gate on `signature_valid IS TRUE`, **NOT** `agent_signature IS NOT NULL`
  — the daemon was signing fine the entire 13-day inert window; only the
  backend verifier was dead, so `agent_signature IS NOT NULL` would have
  falsely read "clean."
- **Zero** open `daemon_heartbeat_signature_unverified`,
  `daemon_heartbeat_signature_invalid`, `daemon_heartbeat_unsigned`
  substrate violations at sign-off.

**Status:** Earliest soak-clean date **2026-05-20** (fix deployed
2026-05-13 14:20 EDT = day 0). v2.0 target 2026-06-03 → ~13 days margin;
not on the critical path. Current prod (2026-05-14): 2897/2897 = 100%
`signature_valid IS TRUE`.

**Soak-check SQL** (per-appliance per-day — the per-appliance floor is
the bar, NOT a fleet aggregate, so `appliance_id` must be in the SELECT):
```sql
SELECT date_trunc('day', observed_at) AS day,
       appliance_id,
       COUNT(*) AS total,
       COUNT(*) FILTER (WHERE signature_valid IS TRUE) AS verified_true,
       round(
         COUNT(*) FILTER (WHERE signature_valid IS TRUE)::numeric
         / NULLIF(COUNT(*), 0), 4) AS verified_ratio
  FROM appliance_heartbeats
 WHERE observed_at > NOW() - INTERVAL '7 days'
   AND appliance_id IN (
     SELECT appliance_id FROM site_appliances
      WHERE agent_public_key IS NOT NULL)
 GROUP BY 1, 2
 ORDER BY 2, 1;
```
Soak is clean only if EVERY (appliance_id × day) row has
`verified_ratio >= 0.99` across the full trailing 7 days.

**Backstop:** `tests/test_baa_artifacts_no_heartbeat_verification_overclaim.py`
fails CI if "heartbeat" co-occurs with verification language in any
customer-facing artifact — pins today's correct scoping against a future
copy regression.
