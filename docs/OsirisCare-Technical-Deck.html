Why This Matters

✓ Auditor Independence

Auditors verify compliance state using Bitcoin proofs — no trust in OsirisCare infrastructure required. The proof is mathematically independent of the platform.

🔗 Append-Only Integrity

Hash chain means modifying any bundle invalidates all subsequent bundles. WORM storage prevents deletion. Combined: tamper-evident by construction.

🛡 Savepoint Isolation

Each proof upgrade runs in a PostgreSQL SAVEPOINT. One failed proof doesn't poison the batch transaction — handles PgBouncer prepared statement edge cases.

LLM Integration: Constrained, Auditable, Self-Reducing

L2 Planner Architecture
# Input: PHI-scrubbed incident metadata { "incident_type": "service_stopped", "check_type": "dns_service", "severity": "high", "hostname": "DC01", "details": "DNS Server service not running" } # Output: runbook selection (NOT arbitrary commands) { "runbook_id": "RB-AUTO-SERVICE_RESTART", "confidence": 0.87, "reasoning": "DNS service critical for AD...", "escalate": false }

🛡 Guardrails

  • Selection only: LLM picks from 88 pre-approved runbooks. Cannot generate commands.
  • Confidence gate: ≥0.6 required for auto-execution. Below → L3.
  • PHI scrubbed: 12-pattern regex runs on-appliance before data reaches API.
  • Runbook steps: Each runbook has explicit WinRM/SSH commands. LLM doesn't write them.
  • Audit trail: Every L2 decision logged in l2_decisions table with full reasoning.

↻ Data Flywheel

When an L2 pattern succeeds 5+ times with 90%+ success rate, it's automatically promoted to an L1 deterministic rule. The system continuously reduces its own AI dependency.

Promoted rules: L1-AUTO-{INCIDENT_TYPE} — tracked in aggregated_pattern_stats + learning_promotion_candidates.

Selection Problem, Not Generation

Deterministic Infrastructure: NixOS Appliance

07
Why NixOS

⚙ Reproducible by Construction

Every appliance is built from the same flake.nix + flake.lock. The Nix store is content-addressed: two identical inputs always produce the identical binary output. This is cryptographic proof of configuration.

🔒 systemd Hardening

  • ProtectSystem=strict — read-only filesystem
  • PrivateTmp=true — isolated temp directories
  • nftables pull-only firewall (no inbound except established)
  • SSH key-only authentication (no passwords)

📦 Golden Flake Deployment

  1. nix build .#appliance-iso — build installer
  2. Write to USB, boot target hardware
  3. Auto-installs, auto-partitions (ESP + MSP-DATA + root)
  4. First boot: phones home to Central Command
  5. MAC lookup provisions identity + API key
  6. Fleet orders deliver config updates remotely

Fleet Order System

✍ Ed25519 Signed Orders

Every fleet order (config update, binary deploy, NixOS rebuild) is cryptographically signed. Go daemon verifies signature before execution. Nonce replay protection with 24h eviction window.

PHI Boundary & Security Controls

08
Data Classification Zones
✅
System Metadata — Always Ingested
Syslog, SSH events, package hashes, backup status, firewall rules, disk encryption state, service status. No patient identifiers.
⚠️
Application Logs — PHI Scrubbed On-Appliance
EHR audit logs, access events. 12-pattern regex strips SSN, MRN, names, DOB, credit cards, phone, email, insurance ID, drug names, ICD-10/CPT codes, lab values, diagnosis text. Matched → [REDACTED] + one-way hash.
🛑
Clinical Data — Never Transmitted
Patient records, diagnoses, prescriptions, lab results, treatment plans. Architecturally excluded at the appliance layer. Cannot reach Central Command.
Key Insight for AI Safety

The Claude API receives incident metadata only: check types, severity levels, hostnames, timestamps. Even if PHI scrubbing fails, the data flowing to the LLM is system operational context — not clinical data. This is defense-in-depth: architectural exclusion + active scrubbing + metadata-only API input.
